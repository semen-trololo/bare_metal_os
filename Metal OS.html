Поток выполнения при старте

  1. GRUB читает заголовок Multiboot (0x1BADB002) из первых 8 КБ файла.
  2. Переключает CPU из 16-бит Real Mode в 32-бит Protected Mode и передаёт управление на _start.
  3. EAX = 0x2BADB002, EBX — адрес структуры Multiboot Information.
  4. Настройка стека ядра (256 КБ, stack_top, выравнивание 16 байт по System V ABI).
  5. Инициализация GDT → IDT → PIC (remap на INT 32–47) → драйверы.
  6. Переход в kernel_main → запуск shell_run().

День 1 — Загрузка и базовая инфраструктура
Multiboot, ELF-секции, стек, VGA-вывод

Заголовок Multiboot

boot.asm NASM 
section .multiboot
align 4
    dd 0x1BADB002                ; магическое число Multiboot
    dd 0x00000003                ; флаги (align + meminfo)
    dd -(0x1BADB002 + 0x00000003) ; контрольная сумма

section .bss
align 16
stack_bottom:
    resb 262144                 ; 256 КБ стека
stack_top:

section .text
global _start
extern kernel_main
_start:
    mov esp, stack_top           ; стек растёт вниз
    call kernel_main
    cli
.hang: hlt
    jmp .hang

Секции ELF-файла

  • .multiboot — заголовок загрузчика, обязан быть в первых 8 КБ файла.
  • .text — исполняемый код (ASM + C).
  • .rodata — константы и строки (только чтение).
  • .data — инициализированные глобальные переменные.
  • .bss — неинициализированные данные; на диске 0 байт, в RAM резервирует объём.

VGA Text Mode

Базовый режим: 80 × 25, буфер по адресу 0xB8000. Каждый символ — 2 байта: ASCII + атрибут цвета (младшие 4 бита — текст, старшие 4 — фон).

Инструментарий

  • Кросс-компилятор i686-elf-gcc с флагами -ffreestanding и -nostdlib.
  • Линкер-скрипт linker.ld: точка входа, базовый адрес 1 МБ, строгий порядок секций.
  • Запуск: qemu-system-i386 -kernel metal_os.bin.

День 2 — Прерывания, драйверы, Shell
VGA 80×50, GDT, IDT, PIC, клавиатура, командная оболочка

VGA 80×50

Перенастройка регистров CRTC через порты 0x3D4/0x3D5: высота скан-линии 8 пикселей, пользовательский шрифт (256 символов × 8 байт) загружается в Plane 2 (0xA0000).

GDT — Flat Memory Model

Плоская модель памяти: сегменты начинаются с 0 и покрывают все 4 ГБ. Разделение на Code (0x9A) и Data (0x92) обеспечивает аппаратную защиту: в коде нельзя писать, в данных нельзя исполнять. Первый дескриптор — NULL (защита от нулевого указателя).

IDT и ISR

interrupts.asm — ASM-заглушка ISR NASM 
%macro ISR_NOERRCODE 1
  global isr%1
  isr%1:
    push byte 0              ; фейковый код ошибки
    push byte %1            ; номер прерывания
    jmp  isr_common_stub
%endmacro

isr_common_stub:
    pusha                    ; EAX, ECX, EDX, EBX, ESP, EBP, ESI, EDI
    push ds
    push es
    push fs
    push gs
    mov  ax, 0x10           ; kernel data segment
    mov  ds, ax
    mov  es, ax
    call isr_handler        ; C-обработчик
    pop  gs
    pop  fs
    pop  es
    pop  ds
    popa
    add  esp, 8              ; чистим err + int_no
    iret

PIC: Remap на INT 32–47

Исторически PIC 8259A шлёт IRQ 0–15 на векторы 0x08–0x0F, занятые исключениями CPU. Через ICW1–ICW4 перенастраиваем на безопасные векторы 32–47. После обработки аппаратного IRQ обязательно шлём EOI (0x20), иначе PIC заблокирует следующие прерывания.

Клавиатура PS/2 и Shell

Архитектура Producer-Consumer: драйвер клавиатуры больше не рисует на экран. Он транслирует скан-коды (Set 1) в ASCII и кладёт их в Ring Buffer. Shell неблокирующе читает через k_getchar(). Decoupling.
  • IRQ1 → INT 33, порты 0x60 (данные) / 0x64 (статус).
  • Машина состояний: Shift, Ctrl, CapsLock с правильной инверсией регистра.
  • Переменные head/tail буфера — volatile.
  • Shell: редактирование строки, k_strcmp для парсинга, hlt в ожидании.

k_printf: конечный автомат

Прямой вывод без промежуточного буфера (динамической кучи пока нет). Парсер идёт по строке формата: обычные символы — в vga_putc(), при % — читается спецификатор.

  • %d — знаковое десятичное (k_itoa).
  • %x — hex с авто-префиксом 0x (k_uitoa).
  • %s — строка (NULL-safe).
  • %c — символ, %% — экранирование процента.

Конвертация чисел

klib.c — k_itoa C 
void k_itoa(int value, char* buf, int base) {
    char tmp[32];
    int i = 0, negative = 0;

    if (value == 0) { buf[0] = '0'; buf[1] = '\0'; return; }

    if (value < 0) { negative = 1; value = -value; }

    while (value > 0) {
        int rem = value % base;
        tmp[i++] = rem < 10 ? rem + '0' : rem - 10 + 'a';
        value /= base;
    }
    /* инверсия: цифры идут от младших к старшим */
    int j = 0;
    if (negative) buf[j++] = '-';
    while (i > 0) buf[j++] = tmp[--i];
    buf[j] = '\0';
}

PIT 8253: честные миллисекунды

timer.c — настройка PIT на 1000 Гц C 
volatile uint32_t tick_count = 0;

void timer_init(uint32_t frequency) {
    outb(0x43, 0x36);        /* канал 0, Lo/Hi, режим 3 */
    uint32_t divisor = 1193182 / frequency;
    outb(0x40, divisor & 0xFF);
    outb(0x40, (divisor >> 8) & 0xFF);
}

void k_sleep(uint32_t ms) {
    uint32_t target = tick_count + ms;
    while (tick_count < target)
        asm volatile("hlt");  /* энергосбережение */
}
⚠️ Wrap-around: при частоте 1000 Гц 32-битный tick_count переполнится через ~49.7 дней. Для долгоживущих систем нужны 64-битные счётчики или функции сравнения с учётом wrap-around.

День 4 — Физическая и виртуальная память
PMM на битмапе, Page Directory/Table, Identity Mapping, MMU

Physical Memory Manager (PMM)

Отслеживание физических страниц по 4 КБ через битмап: 1 бит = 1 страница (0 — свободна, 1 — занята). Для 128 МБ RAM нужно 32768 страниц → битмап 4 КБ.

  • Алгоритм выделения: First Fit (линейный поиск первого свободного бита).
  • Первые 2 МБ зарезервированы (BIOS, VGA, ядро, стек, битмап).
  • При OOM pmm_alloc_page() возвращает 0 — ошибку выводит Shell, а не ядро (Separation of Concerns).

Трансляция виртуального адреса (x86 32-bit)

bits 31–22
Directory Index
10 бит → Page Directory (1024 записи)
bits 21–12
Table Index
10 бит → Page Table (1024 записи)
bits 11–0
Offset
12 бит → смещение в 4 КБ странице

Флаги PDE / PTE (младшие 12 бит)

  • PAGE_PRESENT (бит 0) — страница в RAM; 0 → Page Fault.
  • PAGE_WRITE (бит 1) — разрешена запись.
  • PAGE_USER (бит 2) — доступен из Ring 3.

Identity Mapping и включение MMU

paging.c — включение пейджинга C 
/* Таблицы: выравнивание 4 КБ критично */
uint32_t page_directory[1024] __attribute__((aligned(4096)));
uint32_t page_tables[4][1024] __attribute__((aligned(4096)));

void paging_enable(void) {
    /* CR3 ← физический адрес Page Directory */
    asm volatile("mov %0, %%cr3" : : "r"(page_directory));

    /* Включаем PG (бит 31) и WP (бит 16) */
    uint32_t cr0;
    asm volatile("mov %%cr0, %0" : "=r"(cr0));
    cr0 |= 0x80010000;           /* PG | WP */
    asm volatile("mov %0, %%cr0" : : "r"(cr0));
}
Identity Mapping: виртуальные адреса = физическим. Маппим первые 16 МБ (1 Page Directory + 4 Page Tables), включая EIP, ESP и VGA-буфер 0xB8000. Без этого включение MMU приведёт к Triple Fault: CPU не сможет прочесть следующую инструкцию.
Бит WP в CR0 запрещает ядру писать в Read-Only страницы. Обязателен для защиты памяти и корректной работы User Mode в будущем.

Code Review от ментора
Ключевые замечания и best practices, усвоенные за 4 дня
1
volatile — обязательный Переменные, изменяемые из ISR (tick_count, head/tail ring buffer), обязаны быть volatile. Иначе компилятор закэширует значение в регистре, и циклы ожидания станут бесконечными.
2
aligned(4096) для таблиц страниц Page Directory и Page Tables без атрибута __attribute__((aligned(4096))) сломают флаги PDE/PTE: младшие биты адреса пересекутся с Present/Write/User → Triple Fault.
3
Write Protect (WP) в CR0 При включении MMU нужно устанавливать бит WP (16). Это запрещает ядру (Ring 0) писать в Read-Only страницы — фундаментальная защита памяти.
4
Разделение k_print и k_printf k_print быстрее и не уязвим к инъекциям форматной строки при выводе пользовательского ввода. k_printf — только там, где действительно нужны спецификаторы.
5
Separation of Concerns в ядре Функции ядра (pmm_alloc_page) не должны вызывать UI (k_printf) при ошибках. Возвращают код ошибки (0/NULL) — выводит уже Shell. Это предотвращает спам в логах при фоновых аллокациях.
6
Большие буферы → .bss, а не стек Массив на 32768 адресов для стресс-теста PMM объявляется static или глобально. Иначе стек на 256 КБ переполнится.
7
rep movsw в vga_scroll Сдвиг строк VGA-буфера циклом for неэффективен. Правильно — rep movsw / memmove, аппаратно оптимизированная копия памяти.
8
k_sleep принимает миллисекунды Частота 1000 Гц делает преобразование тривиальным: target = tick_count + ms. Никаких целочисленных делений и потери точности.

Итоговые выводы
Чему научился за 4 дня разработки ОС с нуля
  1. Загрузка без собственного bootloader — разумный компромисс. Multiboot 1 + GRUB снимают с нас 16-битный Real Mode и переключение в Protected Mode, позволяя сосредоточиться на архитектуре ядра.
  2. Кросс-компилятор — не роскошь, а необходимость. i686-elf-gcc с -ffreestanding и -nostdlib изолирует ядро от host-окружения.
  3. Прерывания — нервная система ОС. Без GDT, IDT и корректного remap PIC железо остаётся немым. Стек в ISR обязан быть унифицирован (pusha + сегменты).
  4. Decoupling — архитектурный принцип. Разделение драйвера клавиатуры (producer) и Shell (consumer) через Ring Buffer делает систему тестируемой и расширяемой.
  5. volatile спасает от оптимизатора. Любая переменная, меняемая из ISR, должна быть volatile — это не пожелание, а физическая необходимость.
  6. Paging — момент, когда ОС становится настоящей. Identity Mapping + CR3 + CR0.PG — и 4 ГБ виртуального пространства становятся нашими. Но только при выравнивании таблиц по 4 КБ и установке WP.
  7. Separation of Concerns работает и на уровне ядра. Ядро возвращает коды ошибок, Shell их выводит. Фоновые аллокации не засоряют UI.
  8. Git — не формальность. Снимки каждого этапа позволяют откатиться к работающему состоянию, когда Triple Fault ломает сборку на неизвестной правке.
